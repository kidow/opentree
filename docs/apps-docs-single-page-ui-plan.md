# apps/docs 싱글페이지 UI 레이아웃 개편 계획

## 1. 배경

현재 `apps/docs`는 다음과 같은 구조를 가진다.

- 홈(`app/page.tsx`)은 소개형 랜딩 페이지 역할을 한다.
- 실제 문서 소비는 `/docs/[...slug]` 기반의 다중 페이지 구조로 분리되어 있다.
- 문서 본문은 `content/docs/**/*.mdx`에서 읽어와 렌더링한다.
- 좌측 사이드바, 우측 TOC, 중앙 본문으로 이루어진 전형적인 문서 레이아웃을 사용한다.

이 구조는 일반적인 기술 문서에는 적합하지만, `opentree`의 핵심 가치인 다음 흐름을 직관적으로 보여주기에는 무겁다.

- CLI 명령으로 링크 페이지를 조립한다.
- 조립된 결과가 실제 링크트리 UI로 어떻게 보이는지 바로 확인한다.
- 사용자는 여러 페이지를 이동하지 않고 한 화면에서 전체 흐름을 파악한다.

따라서 `apps/docs`는 더 이상 “문서를 페이지별로 나열하는 사이트”가 아니라, “설명과 결과를 동시에 보여주는 단일 화면형 제품 문서”로 재설계할 필요가 있다.

## 2. 목표

이번 개편의 목표는 다음과 같다.

- 문서와 프리뷰를 하나의 싱글페이지에서 처리한다.
- 데스크탑에서는 좌측 문서 / 우측 플레이그라운드의 2분할 구조를 사용한다.
- 모바일에서는 플레이그라운드를 상단, 문서를 하단에 배치한다.
- 문서 내용은 우선 전부 한 페이지에 몰아넣고, 세부 편집은 후속 작업으로 미룬다.
- 문서 영역은 조립식 블록 구조를 사용하고, 각 블록은 대표 CLI 명령어 한 줄을 가진다.
- 플레이그라운드는 문서 블록이 설명하는 상태를 실제 `opentree` 결과물에 가깝게 미리보기 한다.
- `Next.js`는 유지하되, MDX 기반 다중 페이지 문서 전달 방식은 최대한 제거한다.

## 3. 핵심 UX 원칙

### 3.1 한 화면 중심
사용자는 여러 문서 페이지를 이동하지 않는다.  
문서의 시작점, 사용 흐름, 예시 결과, 핵심 레퍼런스는 모두 하나의 화면 안에서 소비된다.

### 3.2 설명과 결과의 동기화
왼쪽 문서 블록은 단순 설명이 아니라 오른쪽 프리뷰 상태를 바꾸는 역할을 가진다.  
즉, 사용자는 “이 명령이 어떤 결과를 만드는지”를 같은 화면에서 즉시 이해할 수 있어야 한다.

### 3.3 CLI 중심 전달
각 블록에는 대표 CLI 명령어를 한 줄로 고정 노출한다.  
문단이 길어지더라도 사용자는 그 블록에서 기억해야 할 핵심 명령을 바로 볼 수 있어야 한다.

### 3.4 제품형 문서 경험
전형적인 문서 사이드바/TOC 중심 정보 탐색보다, 제품 소개와 조작 데모가 결합된 경험을 우선한다.  
즉, 문서는 “읽기 전용 지식 저장소”가 아니라 “결과를 같이 보는 인터랙티브 설명서”에 가깝다.

## 4. 목표 레이아웃

### 4.1 데스크탑 레이아웃
데스크탑에서는 화면을 좌우 2분할한다.

- 왼쪽: 문서 레이아웃
- 오른쪽: 플레이그라운드 레이아웃

의도는 다음과 같다.

- 왼쪽은 설명과 명령 중심
- 오른쪽은 결과와 시각적 확인 중심
- 사용자는 스크롤하면서 설명을 읽되, 오른쪽에서 결과 맥락을 계속 유지한다

오른쪽 플레이그라운드는 가능하면 sticky 또는 viewport 고정 성격을 가지도록 설계한다.  
왼쪽 문서 영역만 주로 길어지는 구조를 기본값으로 본다.

### 4.2 모바일 레이아웃
모바일에서는 세로 스택으로 전환한다.

- 상단: 플레이그라운드
- 하단: 문서

오른쪽이 위로 오고 왼쪽이 아래로 오는 이유는 다음과 같다.

- 작은 화면에서는 결과를 먼저 보여주는 편이 문서 진입 장벽이 낮다.
- 이후 사용자가 아래로 스크롤하며 설명을 따라갈 수 있다.

### 4.3 제거 또는 축소 대상
현재 구조의 다음 요소는 1차 개편에서 제거하거나 약화한다.

- `/docs/[...slug]` 중심의 독립 문서 페이지 경험
- 전통적 좌측 사이드바 내비게이션
- 우측 TOC 중심 구조
- 홈과 문서가 분리된 이중 진입 흐름
- MDX 페이지 단위 렌더링에 강하게 결합된 문서 프레임

## 5. 정보 구조

싱글페이지는 아래 순서의 서사를 기본 흐름으로 가진다.

1. 제품 한 줄 소개와 문서-플레이그라운드 구조 소개
2. 설치와 시작
3. 기본 프로필 구성
4. 링크 추가와 편집
5. 템플릿/테마 반영
6. 빌드와 로컬 프리뷰
7. 배포 또는 운영 흐름
8. 문제 해결 / 레퍼런스 / 보조 정보

이 순서는 기존 문서 섹션을 그대로 복제하는 것이 아니라, 사용자가 실제로 따라가기 쉬운 단일 흐름으로 재배치한 것이다.

기존 `getting-started`, `guides`, `reference`, `operations`, `examples`는 정보 원천으로는 유지할 수 있지만, 최종 사용자 노출은 하나의 홈 스토리라인으로 통합한다.

## 6. 문서 블록 모델

문서 영역은 “조립식 블록” 단위로 구성한다.  
각 블록은 독립적으로 설명 가능해야 하며, 동시에 우측 프리뷰를 갱신할 수 있어야 한다.

각 블록은 최소한 다음 속성을 가진다.

- `id`: 앵커 및 상태 연결용 식별자
- `title`: 블록 제목
- `summary`: 짧은 설명
- `command`: 대표 CLI 명령어 한 줄
- `body`: 본문 설명
- `previewPatch`: 프리뷰 상태를 어떻게 바꾸는지 정의하는 데이터
- `notes` 또는 `meta`: 후속 확장용 선택 필드

예시 개념은 다음과 같다.

- 설치 블록: `npm install -g opentree-cli`
- 초기화 블록: `opentree init --name "Kidow" --bio "CLI-first profile"`
- 링크 추가 블록: `opentree link add --title "Docs" --url "https://example.com/docs"`
- 템플릿 변경 블록: `opentree theme set ...` 또는 `opentree init --template terminal`
- 배포 블록: `opentree deploy --prod`

### 6.1 CLI 노출 규칙
CLI 명령은 전역 하나를 고정하는 방식이 아니라, 블록별 대표 명령 한 줄을 고정한다.

즉:

- 사용자가 현재 보고 있는 블록마다 핵심 명령이 존재한다.
- 블록이 바뀌면 보여주는 명령도 바뀐다.
- 한 블록 안에서는 여러 명령을 장황하게 나열하기보다, “이 블록을 대표하는 명령 한 줄”을 우선한다.

이 규칙은 문서 전달을 단순하게 만들고, 사용자가 블록 단위로 기억하기 쉽게 만든다.

## 7. 플레이그라운드 설계

### 7.1 방향
플레이그라운드는 docs 전용 가짜 목업이 아니라, 실제 `opentree`가 생성하는 결과물에 최대한 근접해야 한다.

참조 기준은 현재 `src/build.js`가 생성하는 결과 구조다.

핵심적으로 프리뷰는 다음 요소를 재현해야 한다.

- 프로필 카드
- 아바타 또는 이니셜 플레이스홀더
- 이름과 bio
- 링크 카드 리스트
- 템플릿 느낌
- 주요 색상 반영

### 7.2 1차 범위
첫 버전에서 모두 구현할 필요는 없다.  
우선순위는 다음과 같다.

- 프로필 정보 반영
- 링크 리스트 반영
- 테마 색 반영
- `glass` / `terminal` 템플릿 분위기 반영

후순위는 다음과 같다.

- QR 코드
- analytics 표시
- metadata/social card 보조 표현
- 실제 HTML 산출물의 모든 세부 CSS 재현

### 7.3 프리뷰 상태 모델
플레이그라운드는 문서 블록을 따라 변하는 단일 상태를 가진다.

최소 상태 예시는 다음과 같다.

- `profile.name`
- `profile.bio`
- `profile.avatarUrl`
- `links`
- `theme.accentColor`
- `theme.backgroundColor`
- `theme.textColor`
- `template`

이 구조는 `opentree.config.json` 계약과 최대한 정렬되도록 유지한다.  
다만 docs 전용 프리뷰는 전체 스키마를 전부 다루지 않고, 핵심 UI에 필요한 부분집합으로 시작한다.

### 7.4 블록과 프리뷰의 연결 방식
각 문서 블록은 `previewPatch`를 통해 현재 프리뷰 상태를 변경한다.

예를 들어:

- 설치 블록은 프리뷰를 바꾸지 않을 수 있다.
- 초기화 블록은 기본 profile/name/bio/template를 세팅한다.
- 링크 추가 블록은 `links` 배열에 항목을 추가한다.
- 테마 블록은 theme 값을 바꾼다.

이 방식의 장점은 다음과 같다.

- 문서 설명과 프리뷰 변경 로직을 같은 데이터 구조에 묶을 수 있다.
- 실제 페이지 구현 시 상태 전환이 단순해진다.
- 향후 블록 추가/삭제가 쉬워진다.

## 8. 콘텐츠 소스 전략

### 8.1 MDX 제거 방향
이번 개편에서는 기존 MDX 중심 구조를 주 전달 방식으로 보지 않는다.

이유는 다음과 같다.

- 싱글페이지 조립식 블록 경험과 MDX 페이지 단위 구조가 잘 맞지 않는다.
- 문서가 “정적 페이지 묶음”보다 “상태 기반 안내 흐름”에 가까워진다.
- 현재는 문서 섹션을 화면 단위보다 파일 단위로 관리하고 있어, UX 설계 유연성이 떨어진다.

따라서 새 구조의 주 데이터 소스는 MDX가 아니라 더 가벼운 TS/JSON 기반 블록 데이터로 두는 방향을 기본안으로 삼는다.

### 8.2 기존 문서의 역할
기존 `content/docs/**/*.mdx`는 즉시 삭제 대상이라기보다 마이그레이션 참고 소스로 본다.

즉:

- 기존 문서는 새 싱글페이지 블록 문구를 만드는 재료로 사용한다.
- 최종 렌더링은 MDX 페이지 기반이 아니라 블록 기반 렌더링으로 바뀐다.
- 기존 문서 파일을 유지할지 제거할지는 후속 정리 단계에서 결정한다.

## 9. 라우팅 전략

### 9.1 주 진입점
정식 진입점은 홈 싱글페이지로 통일한다.

즉, 사용자가 `apps/docs`에 들어오면 바로 문서+플레이그라운드가 결합된 메인 화면을 보게 한다.

### 9.2 기존 `/docs/*` 처리
기존 `/docs` 및 `/docs/[...slug]`는 더 이상 주요 탐색 구조로 유지하지 않는다.

정리 방향은 다음 중 하나로 수렴하되, 1차 구현에서는 홈 또는 홈 내 앵커 이동 방식이 유력하다.

- 홈으로 리다이렉트
- 대응하는 싱글페이지 섹션 앵커로 이동
- 매우 얇은 호환용 entry route만 유지

핵심 원칙은 다음과 같다.

- 문서 소비의 중심은 더 이상 멀티페이지가 아니다.
- 기존 URL은 가능하면 깨지지 않게 정리한다.
- 새 경험은 한 화면 중심으로 통일한다.

## 10. UI 컴포넌트 구조 제안

싱글페이지는 크게 다음 덩어리로 나뉜다.

- `DocsShell`
- `DocumentColumn`
- `PlaygroundColumn`
- `DocBlockList`
- `DocBlockCard`
- `PinnedCommandBar` 또는 블록 내 명령 카드
- `PreviewFrame`
- `ProfilePreview`
- `LinksPreview`

### 10.1 문서 영역
문서 영역은 긴 설명을 스크롤하는 메인 칼럼이다.

포함 요소:

- 상단 인트로
- 섹션 그룹
- 블록별 제목/설명
- 대표 CLI 한 줄
- 필요 시 짧은 보조 문장
- 앵커 이동 가능 구조

### 10.2 프리뷰 영역
프리뷰 영역은 현재 블록 또는 누적 상태를 기준으로 UI를 렌더링한다.

포함 요소:

- 실제 `opentree`와 유사한 외곽 shell
- 프로필 카드
- 링크 카드
- 템플릿/색상 반영
- 반응형 축소 전략

## 11. 스타일 방향

스타일은 기존 docs 사이트의 장식적 문서 프레임보다, 더 직접적이고 제품적인 인터페이스를 목표로 한다.

원칙은 다음과 같다.

- 문서보다 제품 데모에 가까운 화면
- 좌측 설명과 우측 결과의 대비가 분명한 구조
- 명령어 카드와 프리뷰 카드의 시각적 계층 분리
- 모바일에서도 프리뷰 우선 흐름 유지
- 지나치게 많은 부가 내비게이션 제거

기존 컬러 변수나 일부 질감은 재사용할 수 있지만, 레이아웃 목적이 달라지므로 문서 전용 스타일 체계는 재정비하는 것이 맞다.

## 12. 구현 우선순위

### 12.1 1차 목표
- 홈 싱글페이지 골격 완성
- 좌문서/우프리뷰 반응형 레이아웃 완성
- 블록 데이터 구조 도입
- 블록별 대표 CLI 명령 노출
- 프리뷰 상태 연결
- 기존 주요 문서 내용을 한 페이지 흐름으로 압축

### 12.2 2차 목표
- 블록 간 전환 애니메이션
- 앵커 이동 개선
- 프리뷰와 실제 빌드 산출물 간 시각 차이 축소
- 세부 카피 정리
- 정보 밀도 재조정

### 12.3 후속 과제
- 기존 MDX 파일 정리 또는 폐기 정책 결정
- 기존 `/docs/*` URL의 SEO 및 호환 전략 정리
- 고급 설정/레퍼런스의 싱글페이지 내 배치 최적화

## 13. 테스트 기준

다음 조건을 만족해야 한다.

- 데스크탑에서 좌측 문서 / 우측 프리뷰 2분할이 안정적으로 유지된다.
- 모바일에서 프리뷰가 상단, 문서가 하단으로 쌓인다.
- 문서 블록별 대표 CLI 한 줄이 명확히 드러난다.
- 블록 전환 시 프리뷰 상태가 의도대로 갱신된다.
- 기존 핵심 문서 정보가 한 페이지에 누락 없이 반영된다.
- 기존 `/docs/*` 진입이 홈 중심 구조로 무리 없이 연결된다.
- 프리뷰가 실제 `opentree` 결과 UI와 핵심 인상 면에서 크게 어긋나지 않는다.

## 14. 결론

이번 개편은 단순한 스타일 변경이 아니다.  
`apps/docs`를 “여러 문서를 나열하는 사이트”에서 “CLI 조립 과정과 결과를 동시에 보여주는 제품형 싱글페이지”로 바꾸는 작업이다.

핵심 결정은 다음과 같다.

- `Next.js`는 유지한다.
- MDX 다중 페이지 구조는 주 흐름에서 제거한다.
- 홈 하나에서 전체 문서를 처리한다.
- 문서는 조립식 블록 구조를 사용한다.
- 각 블록은 대표 CLI 한 줄을 가진다.
- 플레이그라운드는 실제 `opentree` UI에 최대한 가깝게 동작한다.

이 문서는 이후 실제 구현 작업에서 레이아웃, 데이터 구조, 라우팅, 프리뷰 설계를 고정하는 기준 문서로 사용한다.
