Opentree publishes by sending your built site to a hosting provider's REST API. You add a token once, the app stores it in your OS Keychain, and every subsequent publish is one click.

Three providers are supported: **Vercel**, **Cloudflare Pages**, and **GitHub Pages**. Pick whichever you already use.

## Where credentials live

Tokens are stored in your operating system's secure credential store via the [`keyring`](https://crates.io/crates/keyring) crate (macOS Keychain). They never leave your machine and are not written to your `opentree.config.json` file.

To add or update a token, open **Settings → Connections**.

## Vercel

### 1. Create a token

1. Go to [vercel.com/account/tokens](https://vercel.com/account/tokens).
2. Click **Create Token**.
3. Scope: account-wide is fine for personal projects. For teams, scope to the team.
4. Expiration: your call. Opentree only uses the token for deploys.
5. Copy the token value (you won't see it again).

### 2. Connect in the app

1. Open **Settings → Connections → Vercel**.
2. Paste the token. The app calls `GET /v2/user` to verify it's valid.
3. If the token is wrong, you'll see "유효하지 않은 Vercel 토큰입니다."

### 3. Publish

1. Open the **Publish** tab.
2. Choose **Vercel** as the provider.
3. Enter a **project name** (Vercel will create the project on first deploy if it doesn't exist).
4. Click **Publish**.

The first publish creates the project and assigns a `*.vercel.app` URL.

## Cloudflare Pages

### 1. Create an API token

1. Go to [dash.cloudflare.com/profile/api-tokens](https://dash.cloudflare.com/profile/api-tokens).
2. Click **Create Token** → **Create Custom Token**.
3. Permissions: **Account · Cloudflare Pages · Edit**.
4. Account Resources: include the account you'll deploy to.
5. Save and copy the token.

### 2. Find your Account ID

In the Cloudflare dashboard, open any zone or the right sidebar of the Pages section — your Account ID is shown there.

### 3. Connect in the app

1. Open **Settings → Connections → Cloudflare Pages**.
2. Paste the **token** and **Account ID**.
3. The app calls `GET /accounts/{account_id}/pages/projects` to verify both are valid.
4. If either is wrong, you'll see "유효하지 않은 Cloudflare 토큰 또는 Account ID입니다."

### 4. Publish

1. Open **Publish → Cloudflare Pages**.
2. Enter a **project name**.
3. Click **Publish**.

Cloudflare Pages assigns a `*.pages.dev` URL on first deploy.

## GitHub Pages

### 1. Create a personal access token

1. Go to [github.com/settings/tokens](https://github.com/settings/tokens).
2. Either token type works — classic or fine-grained.
3. Required scopes:
   - **`repo`** — to push files to the repository.
   - **`workflow`** — only if your repo uses GitHub Actions for Pages.
4. Copy the token.

### 2. Pick or create a repository

You'll need a repository to host the Pages site. The app pushes built files to its default branch. Create a public empty repo on GitHub if you don't already have one.

### 3. Connect in the app

1. Open **Settings → Connections → GitHub Pages**.
2. Paste the **token** and the `owner/repo` (e.g. `kidow/my-bio`).
3. The app calls `GET /user` to verify the token.
4. If the token is wrong: "유효하지 않은 GitHub 토큰입니다."

### 4. Publish

1. Open **Publish → GitHub Pages**.
2. The app pushes built files via the GitHub Contents API and activates the Pages site.
3. The published URL is `https://<owner>.github.io/<repo>/`.

## What gets uploaded

A publish always includes:

- The generated `index.html`
- The `assets/` folder (avatar, images you imported into the app)
- Any CSS/JS the renderer emits

Nothing is uploaded that isn't shown in the preview.

## Switching providers

Connections are independent — you can have all three connected at once and pick a provider per publish from the **Publish** tab. Disconnecting a provider only removes its token from the Keychain; the published site stays live until you delete it from that provider's dashboard.
